# Project Context Records (PCR)

[English](./README.md) · **简体中文**

> 一套面向 AI 时代的方法论：在仓库中维护一份持久的、随代码一起版本化的项目背景记录——包括*为什么这样
> 做*、架构、维护者做出的决策、以及背后的意图——使得一个没有长期记忆、又会自行判断的 AI 协作者能够
> **继承**项目已有的判断，而不必从头重新推导。

PCR 是 **context engineering（上下文工程）** 的一种具体实践。Context engineering 关注"应该把哪些内容放进
模型的上下文"；PCR 则是其中**属于某个项目、并需要长期保存**的那一部分——值得版本化、值得审阅、值得在
不同会话之间传承下去的内容。

## TL;DR：如何使用

把下面这段内容粘贴到你项目的 `AGENTS.md` / `CLAUDE.md` 中（并把路径改成你实际存放记录的目录）：

```markdown
## Project Context Records (PCR)

This project follows **Project Context Records (PCR)** —
methodology: https://github.com/hyf0/project-context-records. PCR keeps the
project's durable design context — the *why*, the decisions, the architecture —
so you inherit it instead of re-deriving or re-litigating what's already settled.

When working here:
- **Where they live.** Records are in `.agents/docs/` — one concept per file,
  cross-linked with relative Markdown links (`[name](./name.md)`).
- **Read first.** If a record covers the area you're touching, read it before acting.
- **Record as you go.** Proactively write down context worth keeping — and whenever
  a human asks you to. No required format, no fixed list of what qualifies: if it's
  true about this project but not visible in the code, it's worth a record.
- **Keep it fresh.** If your change affects a record, update it in the same change —
  a stale record is a trap, not an asset.
- **Provenance.** An unstamped line is AI-accumulated: challenge and verify it freely.
  A line stamped `[✓ @handle]` was vouched for by a human — don't reopen or
  re-verify it. Add a stamp only when a human explicitly tells you to; a human
  reading past a line, or not objecting, is not a stamp.
```

这一段就是采纳 PCR 的全部内容。它把每一位协作者——无论是人还是 agent——引导到记录所在的位置，并写明
需要遵守的规则。它**有意写得很薄**：agent 每个会话都会自动读取 `AGENTS.md`，但不会主动去获取一个 URL，
因此只把"每个会话都必须遵守"的内容写进来；其余内容留在一个链接之后，agent 不需要获取它也能正确工作。

之后正常开展工作即可：在工作过程中**持续积累（accumulate）**记录，并在有人审阅时对它们进行
**提炼（distill）**——把有价值的内容保留并提升，把过期的内容删去，对经过人确认的内容加上标记。

**以下全部是 *why*（原理与缘由）。**

---

## 前提：模型的输出取决于它的上下文

模型的输出是其上下文的函数。如果给它提供决定答案所需的上下文，结果就会更接近项目真正想要的样子；如果
缺少这部分上下文，它就会用看似自信的猜测来填补空缺——而那些看似合理、实则错误的结果，正是这样产生的。

关键在于这里存在一种**不对称**：上下文过多是有代价的（会引入噪音、分散注意力、消耗 token）；但缺少
决定答案的那部分上下文却没有任何好处，只会让结果变差。一个能力很强的模型，在缺少关键信息时同样会得出
错误的结果；它能达到的上限，取决于它能看到的信息。

正因如此，PCR 包含两个部分：**积累（accumulate）**那些原本会随会话消失的判断（上下文太少必然有害），以及
**提炼（distill）**它们——删去噪音、保持当前内容的准确（上下文过多或过期同样有害）。

## 为什么仅有代码已经不够

代码记录的是**最终状态**：系统最后呈现出来的样子。它在结构上丢失了协作者需要的三样东西：

1. **被放弃的方案。** 代码只显示"我们选择了 A"，却不会显示"我们也考虑过 B 和 C，B 因为某个原因被
   否决"。于是下一位协作者看到 A，会问"为什么不用 B"，然后真的去尝试 B，最终又一次遇到当初同样的
   障碍。
2. **某个选择的状态。** 这是经过深思熟虑、已经定下的决策，还是一个临时方案、还等待重新审视？代码在
   两种情况下看起来完全一样，但它们要求采取的行动恰好相反。
3. **某个判断的来由。** 它建立在哪些约束、哪些权衡、由谁决定之上？当约束发生变化时，这个决策可能需要
   重新考虑——但前提是当初记录下了它所依赖的条件。

在 AI 出现之前，这些内容存在于维护者的头脑中，通过**连续性**来传递：代码审阅、经验传授、"去问当初写它
的人"。团队是连续的，知识虽然是隐性的，但不会中断。

而 AI 协作者**没有连续性**。每个会话都是一位没有记忆、能力很强、又会自行判断的新人。它不会"去问当初
写它的人"，而是自己推理、自己怀疑、自己动手。隐性知识对它来说等于不存在。因此，那一直存在、却藏在
人脑中的判断层，现在必须被固化成**明确、可见的内容**——否则它会在每个会话中重新消失一次，反复如此。

PCR 就是用来承载这些内容的。它保存的不是*代码本身是什么*，而是*这些判断是如何得出的*——让已经付出过
的思考，不必再重新付出一遍。

## 老树发新枝

PCR 是 **Architecture Decision Records（ADR，架构决策记录）** 在 AI 时代的延续——ADR 这套做法由 Michael
Nygard 于 2011 年提出，后来发展成一个[包含工具与模板的生态][adr-hub]。一条 ADR 把每个重要决策记成一个
小的、带编号、纳入版本管理的文件，写明它的*背景（context）*、*决策（decision）*和*影响（consequences）*；
一条已经定下的记录不再修改——要改变它，就新写一条来**取代（supersede）**它，从而留下一份只增不改的轨迹。

PCR 保留了 ADR 最好的一点——*知识存放在仓库里，和代码一起审阅、一起版本化，而不是放进 wiki*——并针对
新的读者补充了三点：

1. **从决策扩展到上下文。** ADR 记录的是一次性的选择。PCR 记录的是整个背景层：决策，以及架构、思路、
   有意的偏离、易错点、还有正在进行中的计划。一次性决策只是项目背景的一部分，而非全部。
2. **从"不可修改、层层叠加"到"单一、保持最新"。** ADR 通过冻结每个已定文件、再追加一条取代它的文件
   来保存历史，结果当前的真实情况散落在一条取代链上。PCR 则把**当前结论放在同一处并保持最新**，把演变
   过程交给 **git** 来保存。让一个没有记忆的 agent 为了弄清今天的答案而重读整条取代链，是一种浪费；不如
   直接给它一个可靠的答案，把历史留给版本记录。
3. **读者从人变成会自行判断、且没有记忆的 agent。** ADR 只做描述，因为人读完影响后会自我约束。而 agent
   会"发现"一件已经定下的事，然后直接动手。因此一条 PCR 记录可以带上**给 agent 的明确指示**（"这一项
   不在对外约定之内，不要再去改动它"），而不只是单纯的描述。

## 记录哪些内容

一个项目的**背景层**：它为什么是现在这个样子、架构、维护者的决策、意图、有意的偏离、思路、来之不易的
经验教训，以及当前正在进行的计划。

换一个角度，往往更清楚：**任何关于这个项目为真、却无法从代码中看出来的内容。**

记录时应当**放宽**——只要有价值，就随手、低成本地记下来。是否有价值，留到之后的**提炼**环节（见下）再
筛选，而不是在记录时就筛。临时性的内容（一份实现计划、一段中途的思考）可以暂时记着；经过提炼后留下来
的，才是真正有价值的部分。

## 机制：盖章（provenance stamp）

这是 PCR 唯一的核心机制，它建立在一条基本原则之上：

> 在 AI 时代，文字是廉价的，而且**默认由 AI 写出**。真正稀缺、真正有价值的动作，是**由一个人对某条
> 内容表示负责、为它背书。**

因此规则只有一条：

- **没有盖章 = 由 AI 积累。** 这是默认内容，应当把它当作可以质疑的——尽管去质疑、去验证。
- **带有 `[✓ @handle]` 盖章 = 已由某个人审阅并为其背书。** 不要重新翻动它，也不要浪费精力反复验证，把它
  当作可靠的基础。盖章是一个明确的动作：由人有意加上——人只是扫过某条内容、或者没有提出反对，都不算
  盖章。

```
时间戳一律以 UTC 存储，仅在边界处转换 —— 在一次夏令时（DST）问题之后定下。   [✓ @hyf0]

把缓存有效期（cache TTL）调到 1 小时大概是安全的。   ← 没有盖章：由 AI 积累，请自行验证
```

**人的参与是可选的——它是一种质量上的提升，而不是必需的组成部分。** 由 AI 积累的内容本身就能用；人的
盖章只是把其中一部分内容**提升**一档可信度。把人有限的精力花在风险较高、值得花的地方，其余的就作为
诚实的默认状态留着。

**为什么要盖章，又为什么只有一种。** 一个由 AI 维护的知识库，最严重的失败是：AI 自己的猜测被当成了
人已经确认的事实——后来的 agent 把一个由机器做出的假设当作已经定下的结论，并在其上继续构建。盖章
正是为了划出这一条界线，而且只划这一条：**是否经过人的背书。** 更细的区分——*这是决策还是观察？是
已定还是暂定？*——应当写进记录本身的正文里，而不是增加更多种类的盖章。先从这一种开始；只有当实际
使用证明确有需要时，再增加。

## 工作流程：积累 → 提炼

- **积累（accumulate）**（成本低、默认进行、由 agent 完成）：在工作过程中主动记录上下文——以及在人提出
  要求时记录——包括临时的计划和中途的思考。数量多没关系，有噪音也是预期之中的。
- **提炼（distill）**（成本高、由人完成）：一次审阅，把有价值的内容**保留并提升**，把错误和过期的内容
  **删去**，对审阅者愿意背书的内容**加上盖章**。提炼是让低成本的积累变成持久、可信记录的途径——它是把
  "人的确认"作为一项常规工作来执行，也正是"临时记录的实现计划"转变为"真正留下来的有价值内容"的环节。

## 写作约定

- **一个概念一个文件。** 用标准的相对 Markdown 链接（`[other-concept](./other-concept.md)`）在相关记录
  之间互相引用——在 GitHub 上可点击、不产生歧义、也不依赖特定工具。
- **保持当前内容的准确。** 内容与现实脱节是最大的风险：一条已经不符合现实、却写得很肯定的记录，会从
  资产变成**陷阱**——agent 会满怀信心地照着它去做。一条过期的记录比没有记录更糟。当你的改动影响到某条
  记录时，在同一次改动中一并更新它。
- **写清楚 why（为什么）**，而不只是 what（是什么）：权衡了什么、放弃了哪些备选、已知有哪些易错点。先讲
  原则或核心思路。
- **写到让协作者不必读代码内部也能理解。** 每条记录作为独立的文字就能读懂；agent 读完就能理解。
- **不规定固定格式——这是有意的。** 一条记录就是能讲清楚 why 的文字。更严格的格式也许会从实际使用中
  自然形成，但不要一开始就强加。
